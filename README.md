# hatch_project

> 브라우저에서 펫을 부화시키고 돌보는 작은 웹 게임

`hatch_project`는 웹에서 펫을 생성하고, 상태를 확인하고, 여러 액션을 통해 컨디션을 관리하는 다마고치 스타일의 게임입니다. 단순한 화면 데모가 아니라, 펫의 상태가 서버와 데이터베이스에 저장되고 시간이 지나면서 실제로 변화하도록 구성했습니다. 작은 게임이지만 프론트엔드, 백엔드, 데이터 저장, 캐시, 실행 환경까지 한 흐름으로 다뤄보는 데 초점을 두고 개발하고 있습니다.

## 프로젝트 소개

플레이어는 먼저 게스트 사용자를 만들고, 자신의 펫을 생성한 뒤 상태를 확인할 수 있습니다. 이후 `feed`, `clean`, `play`, `sleep` 액션을 수행하면서 배고픔, 청결도, 행복도, 에너지, 건강 수치를 관리하게 됩니다. 시간이 지나면 상태가 계속 변하기 때문에, 버튼만 누르는 화면이 아니라 상태를 지속적으로 관리하는 게임 흐름을 목표로 했습니다.

## 주요 기능

- 게스트 사용자 생성
- 펫 생성
- 펫 상태 조회
- `feed / clean / play / sleep` 액션
- 시간 경과에 따른 상태 변화
- 위험 상태(`sick`)와 사망 상태(`dead`) 반영
- 액션 로그 저장
- 헬스 체크 엔드포인트 제공

## 개발하면서 신경 쓴 점

### 1. 시간 경과를 반영하는 상태 모델
단순히 버튼을 누를 때만 값이 바뀌는 방식보다는, 시간이 지날수록 펫 상태가 자연스럽게 변화하도록 만들었습니다. 조회 시점 기준으로 경과 시간을 계산해 `hunger`, `cleanliness`, `happiness`, `energy`, `health`에 반영하고, 조건에 따라 `alive`, `sick`, `dead` 상태가 바뀌도록 구성했습니다.

### 2. 상태 조회와 상태 변경을 분리한 구조
펫 상태 조회는 별도 API로 두고, 액션 수행 API는 상태를 변경하는 역할만 맡도록 나눴습니다. 작은 게임이지만 요청-응답 구조를 분리해두면 UI와 서버 로직을 각각 다루기 쉬워지고, 이후 기능을 확장할 때도 구조를 유지하기 편하다고 판단했습니다.

### 3. Redis를 활용한 응답 캐시와 액션 제어
상태 조회 API에는 Redis TTL 캐시를 적용하고, 액션이 발생하면 캐시를 무효화하도록 구성했습니다. 또 동일 액션이 짧은 시간 안에 반복 호출될 경우를 제어하기 위해 Redis 기반 rate limiting도 넣었습니다. 게임 버튼 하나라도 서버 입장에서는 요청이기 때문에, 조회와 변경 흐름을 조금 더 안정적으로 다루는 연습을 해보고 싶었습니다.

### 4. 로그와 상태 확인 포인트 추가
백엔드에는 요청마다 request id를 부여하고 처리 시간을 로그로 남기도록 했습니다. 또한 `/api/health/live`, `/api/health/ready` 엔드포인트를 분리해 서비스 상태를 확인할 수 있도록 구성했습니다. 개발 중에는 기능 구현만큼이나, 실행 중인 서비스의 상태를 빠르게 확인할 수 있는 구조도 중요하다고 생각했습니다.

### 5. 실행 환경 통일
로컬 실행 환경은 Docker Compose로 통일했습니다. 프론트엔드, 백엔드, PostgreSQL, Redis를 한 번에 올릴 수 있게 구성해 새 환경에서도 같은 방식으로 실행할 수 있도록 정리했습니다. 별도로 백엔드 테스트는 pytest로 구성했고, GitHub Actions에는 기본적인 backend CI 흐름을 추가했습니다.

## 아키텍처

```text
[ React Frontend ]
        |
        v
[ FastAPI Backend ]
   |            |
   v            v
[PostgreSQL]  [Redis]
```

- Frontend는 플레이어 입력과 상태 표시를 담당합니다.
- Backend는 펫 상태 계산, 액션 처리, API 응답을 담당합니다.
- PostgreSQL은 사용자, 펫, 액션 로그 같은 상태 데이터를 저장합니다.
- Redis는 상태 조회 캐시와 반복 액션 제어에 사용합니다.
